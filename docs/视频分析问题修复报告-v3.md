## 一、问题分析报告

### 1.1 视频分析概要

对屏幕录制视频 `屏幕录制 2026-07-03 151923.mp4` 全面分析，视频时长约 2 分钟，涵盖：

| 阶段 | 时间段 | 内容 |
|------|--------|------|
| 前台访问 | 00:00 - 00:18 | /login 显示首页内容、教材查询（需登录）、校历查询、新闻中心 |
| 登录异常 | 00:15 - 00:30 | 多次点击"登录"按钮，URL 为 /login 但始终显示首页内容 |
| 首页滚动 | 00:19 - 00:30 | 展示"为什么选择我们"、功能卡片、数据统计、新闻资讯等 |
| 管理后台 | 00:31 - 01:00 | 管理员入口 → 仪表盘（6-7s加载）→ 返回前台 → 退出 → 重新登录 |
| 后台模块 | 01:00 - 01:38 | 教材管理、新闻管理、用户管理、校历管理（各 3-5s 加载） |
| 学院管理超时 | 01:39 - 01:56 | 学院管理加载 15s 后超时，红色错误"请求超时，请稍后重试" |

### 1.2 发现的问题清单

| # | 问题 | 严重程度 | 出现场景 | 复现步骤 |
|---|------|----------|----------|----------|
| 1 | 前台 /login 显示首页内容（登录页不可用） | **P0** | 访问 /login | 1. 浏览器中残留旧版 admin token（存储在 `token` key） 2. 访问 /login 3. 路由守卫判断"已登录"→ 重定向首页 4. URL 显示 /login 但内容是首页 |
| 2 | /admin/colleges 请求超时（学院管理不可用） | **P0** | 后台 → 学院管理 | 1. 登录管理后台 2. 点击"学院管理" 3. 加载 15 秒后超时 4. 显示"请求超时，请稍后重试" |
| 3 | 后台各列表页加载缓慢（5-7秒） | **P1** | 后台菜单切换 | 1. 登录管理后台 2. 在仪表盘/教材/新闻等菜单间切换 3. 每次切换 loading 3-7 秒 |
| 4 | 错误提示缺少重试引导 | **P2** | 学院管理超时 | 1. 学院管理加载超时 2. 仅显示错误文字 3. 无重试按钮，用户不知如何继续 |

### 1.3 根因分析

#### 问题1根因：旧版本残留 token

**问题链条**：
1. 用户之前使用旧版本代码（前后台共用 `token` localStorage key）
2. 通过 /admin/login 登录后，admin token 存入了 `token` key，user.role = 'ADMIN'
3. 升级到新版本后（admin 登录改用 `adminToken` key），旧 `token` 仍残留在浏览器
4. 访问 /login → 路由守卫检查 `localStorage.token` 存在 → 判断"已登录" → 重定向到 /
5. URL 显示 /login 但渲染首页内容

**核心矛盾**：代码已修复（独立 adminToken），但用户浏览器的旧数据未清理。

#### 问题2根因：学院管理 API 无缓存 + 冷启动

**原实现**：
- `/api/admin/colleges` 一次性加载所有学院（8个）+ 专业（18个）+ 班级（108个）的嵌套数据
- 无服务端缓存
- Neon Serverless 冷启动时，Prisma 连接建立慢
- 客户端 15 秒超时 → 请求失败

#### 问题3根因：后台列表 API 无缓存

**原实现**：所有后台 GET 列表 API 无服务端缓存，每次菜单切换都重新查询数据库。虽然有 keep-alive 缓存组件，但首次访问仍需等待 API 响应。

#### 问题4根因：错误处理缺少引导

**原实现**：CollegeManageView 错误提示仅显示文字，无重试按钮。

## 二、修复方案

### 2.1 修复方案总览

| # | 问题 | 修复方案 | 优先级 | 状态 |
|---|------|----------|--------|------|
| 1 | /login 显示首页 | 路由守卫识别旧版残留 token + main.ts 启动清理 | P0 | ✅ 已修复 |
| 2 | 学院管理超时 | API 添加 5 分钟缓存 + 客户端超时提升到 30s | P0 | ✅ 已修复 |
| 3 | 后台列表加载慢 | 所有后台 GET API 添加缓存（60-600s 分级） | P1 | ✅ 已修复 |
| 4 | 错误提示无引导 | CollegeManageView 添加重试按钮 | P2 | ✅ 已修复 |

### 2.2 详细修复方案

#### 修复1：路由守卫识别并清除旧版残留 token（问题1）

**文件**：`src/router/index.ts`

**修改内容**：
```typescript
// 已登录前台访问前台登录/注册页 → 跳转首页
// 但如果 user 角色是 ADMIN，说明是旧版本残留的 admin token，清除它并显示登录页
if (isFrontAuthed && (to.name === 'login' || to.name === 'register')) {
  if (isFrontAdmin) {
    // 旧版本残留：admin token 存储在 token key 中，清除并显示登录页
    localStorage.removeItem('token')
    localStorage.removeItem('user')
    next()
    return
  }
  next({ path: '/' })
  return
}
```

**文件**：`src/main.ts`

**修改内容**：启动时自动清理旧版残留 token
```typescript
;(function cleanupStaleTokens() {
  try {
    const userStr = localStorage.getItem('user')
    if (userStr) {
      const user = JSON.parse(userStr)
      if (user?.role === 'ADMIN') {
        localStorage.removeItem('token')
        localStorage.removeItem('user')
        console.info('[cleanup] 已清除旧版本残留的 admin token')
      }
    }
  } catch {
    localStorage.removeItem('user')
    localStorage.removeItem('token')
  }
})()
```

**效果**：
- 用户访问 /login 时，如果检测到旧版残留的 admin token，自动清除并显示登录页
- 应用启动时自动清理旧版数据，无需用户手动清除浏览器缓存
- 前后台登录完全隔离：`token`（前台）vs `adminToken`（后台）

#### 修复2：学院管理 API 添加缓存 + 超时提升（问题2）

**文件**：`api/index.ts`

**修改内容**：
```typescript
// 学院管理列表添加 5 分钟缓存
{ method: 'GET', pattern: '/api/admin/colleges', handler: handleAdminCollegeList, auth: true, cache: 300 },
```

**文件**：`src/utils/request.ts`

**修改内容**：请求超时从 15s 提升到 30s
```typescript
const service: AxiosInstance = axios.create({
  baseURL,
  timeout: 30000,  // 从 15000 提升，应对 Neon 冷启动
  headers: { 'Content-Type': 'application/json' },
})
```

**效果**：
- 首次访问学院管理：API 查询数据库（可能 5-10s），结果缓存 5 分钟
- 后续访问：从缓存返回（<100ms）
- 写操作（新增/编辑/删除）自动失效缓存（`invalidateCache` 已实现）
- 30s 超时足够应对 Neon 冷启动

#### 修复3：所有后台 GET 列表 API 添加缓存（问题3）

**文件**：`api/index.ts`

**缓存策略**：

| API | 缓存时间 | 理由 |
|-----|----------|------|
| /api/admin/stats | 120s | 统计数据变化频率中等 |
| /api/admin/textbooks | 120s | 教材列表，增删改时自动失效 |
| /api/admin/news | 120s | 新闻列表，增删改时自动失效 |
| /api/admin/users | 60s | 用户列表，变化频率较高 |
| /api/admin/semesters | 300s | 学期数据低频变更 |
| /api/admin/colleges | 300s | 学院数据低频变更 |
| /api/admin/courses | 300s | 课程数据低频变更 |
| /api/admin/categories | 600s | 新闻分类极少变更 |

**缓存失效机制**：所有后台写操作（POST/PUT/DELETE）后，自动调用 `invalidateCache('get:')` 清除所有 GET 缓存，确保数据一致性。

**效果**：
- 首次访问：正常查询数据库
- 后续访问（缓存有效期内）：从内存缓存返回，<100ms
- 数据变更后：缓存自动失效，下次访问重新查询

## 三、前后台登录分离说明

### 3.1 完全分离方案

| 维度 | 前台 | 后台 |
|------|------|------|
| 登录 URL | `/login` | `/admin/login` |
| localStorage token key | `token` | `adminToken` |
| localStorage user key | `user` | `adminUser` |
| Pinia store | `useAuthStore` | `useAdminAuthStore` |
| 路由守卫 | `requiresAuth` | `requiresAdmin` |
| 退出登录 | 清除 `token`/`user` | 清除 `adminToken`/`adminUser` |
| 互不干扰 | ✅ 后台登录不影响前台 | ✅ 前台登录不影响后台 |

### 3.2 路由守卫逻辑

```
访问 /admin/* → 检查 adminToken → 无效 → 跳转 /admin/login
访问 /login   → 检查 token → 有效且 role=ADMIN → 清除残留 token → 显示登录页
访问 /login   → 检查 token → 有效且 role≠ADMIN → 跳转首页
访问 /login   → 检查 token → 无效 → 显示登录页
```

### 3.3 旧版本兼容

- 应用启动时自动检测并清除旧版本残留的 admin token（`token` key 中 role=ADMIN 的数据）
- 路由守卫双重保护：即使启动清理未执行，访问 /login 时也会检测并清除
